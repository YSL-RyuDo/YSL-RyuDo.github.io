UML 1주차: 클래스 다이어그램이 왜 필요한가?

Week 1 Draft

🧩 클래스 다이어그램은 무엇인가?

클래스 다이어그램은 프로그램을 구성하는 클래스들이 어떤 데이터와 기능을 가지고 있는지, 그리고 서로 어떤 관계를 맺는지 보여주는 설계도다.

코드로 바로 보면 수많은 파일과 함수에 흩어져 있는 구조를 하나의 그림으로 압축해서 볼 수 있다.

게임 개발에서는 Player, Weapon, Room, Matchmaker, Session 같은 개념이 계속 등장한다. 클래스 다이어그램은 이런 개념들이 어디에 속하고, 어떤 책임을 가져야 하는지 정리하는 데 도움을 준다.

🛠 코드 작성 전에 설계가 필요한 이유

코드를 먼저 작성하면 당장은 빠르게 움직이는 것처럼 보인다.

하지만 기능이 늘어날수록 어느 클래스가 어떤 일을 해야 하는지 흐려진다. 처음에는 간단했던 이동, 공격, 무기 장착 기능이 나중에는 네트워크 동기화, DB 저장, 매칭, 방 관리와 연결되면서 구조가 복잡해진다.

클래스 다이어그램을 먼저 그려보면 코드를 작성하기 전에 책임을 나눠볼 수 있다.

Player는 플레이어의 상태와 행동을 담당하고, Weapon은 무기 동작을 담당한다. 이런 식으로 역할을 나누면 코드가 커져도 구조를 추적하기 쉬워진다.

💥 코드부터 작성하면 생기는 문제

게임 프로젝트를 처음 만들 때 자주 나오는 실수가 있다.

바로 GameManager 하나에 모든 기능을 넣는 방식이다.

처음에는 GameManager가 플레이어 생성, 이동, 공격, 무기 장착, 데미지 처리까지 모두 담당해도 크게 문제 없어 보인다. 하지만 기능이 늘어나면 GameManager는 점점 거대한 클래스가 된다.

이런 구조에서는 수정할 때마다 다른 기능이 같이 흔들릴 수 있다.

예를 들어 공격 로직을 고치려 했는데 플레이어 생성 코드와 무기 장착 코드까지 같이 확인해야 한다면, 이미 책임이 제대로 분리되지 않은 상태라고 볼 수 있다.

🚫 나쁜 예시: 모든 기능이 GameManager에 몰린 구조

class GameManager
{
    int playerHp;
    Vector3 playerPosition;
    string playerNickname;

    int weaponDamage;
    int weaponAmmo;

    void MovePlayer();
    void Attack();
    void EquipWeapon();
    void TakeDamage();
    void FireWeapon();
    void ReloadWeapon();
}

이 코드는 보기에는 단순하지만, Player의 책임과 Weapon의 책임이 GameManager 안에 섞여 있다.

플레이어가 여러 명이 되거나, 무기 종류가 늘어나거나, 서버에서 여러 방을 관리해야 하는 순간 구조가 빠르게 무너질 수 있다.

게임 서버에서는 이런 문제가 더 크게 느껴진다. 한 명의 플레이어만 관리하는 것이 아니라 여러 세션, 여러 방, 여러 요청을 동시에 다뤄야 하기 때문이다.

🎮 예제: Player와 Weapon으로 나눠보기

플레이어는 체력과 위치를 가진다.

플레이어는 이동하고 공격할 수 있다.

플레이어는 무기를 장착할 수 있다.

무기는 공격력과 탄약 수를 가진다.

무기는 발사와 재장전을 할 수 있다.

이 요구사항을 보면 Player와 Weapon이라는 두 개의 클래스 후보를 뽑을 수 있다.

Player는 플레이어의 상태와 행동을 담당하고, Weapon은 무기의 상태와 동작을 담당한다. Player가 Weapon을 가진다는 관계도 자연스럽게 생긴다.

📐 예제 클래스 다이어그램 설명

클래스 다이어그램은 보통 클래스 이름, 속성, 메서드로 나눠 표현한다.

클래스 이름은 가장 위에 적는다. 여기서는 Player와 Weapon이다.

속성은 클래스가 가지고 있는 데이터다. Player의 hp, position, nickname, weapon이 여기에 해당한다.

메서드는 클래스가 수행할 수 있는 행동이다. Move(), Attack(), EquipWeapon(), TakeDamage() 같은 기능이 여기에 들어간다.

접근 제어자도 함께 표시할 수 있다. UML에서 -는 private, +는 public을 의미한다.

즉, 외부에서 직접 건드리면 안 되는 데이터는 private으로 숨기고, 외부에서 사용할 행동은 public 메서드로 열어둔다.

🧾 코드블럭 정리

Player
--------------------------------
- hp : int
- position : Vector3
- nickname : string
- weapon : Weapon
--------------------------------
+ Move()
+ Attack()
+ EquipWeapon()
+ TakeDamage()

Weapon
--------------------------------
- damage : int
- ammo : int
--------------------------------
+ Fire()
+ Reload()
class Player
{
private:
    int hp;
    Vector3 position;
    string nickname;
    Weapon weapon;

public:
    void Move();
    void Attack();
    void EquipWeapon(Weapon newWeapon);
    void TakeDamage(int damage);
};

class Weapon
{
private:
    int damage;
    int ammo;

public:
    void Fire();
    void Reload();
};

이렇게 나누면 Player는 플레이어에 대한 책임을 가진다.

Weapon은 무기 자체의 책임을 가진다.

GameManager는 모든 일을 떠안는 클래스가 아니라, 필요한 객체들을 연결하거나 전체 흐름을 조율하는 역할에 가까워진다.

🗺 클래스 다이어그램은 예쁜 그림이 아니라 설계도

클래스 다이어그램을 처음 보면 단순히 네모 박스를 예쁘게 그리는 작업처럼 보일 수 있다.

하지만 실제 목적은 그림을 꾸미는 것이 아니다.

어떤 클래스가 어떤 책임을 가지는지, 어떤 데이터가 어디에 있어야 하는지, 클래스끼리 어떻게 연결되는지를 미리 생각하는 것이 핵심이다.

좋은 클래스 다이어그램은 코드 구조와 연결된다. 그림에 있는 책임 분리가 실제 코드에서도 유지되어야 의미가 있다.

🧑‍💻 게임 서버 코드로 생각해보기

Player와 Weapon 예제는 작아 보이지만, 서버 구조로 넘어가면 같은 문제가 더 크게 나타난다.

Player는 단순히 캐릭터 정보만 가지는 것이 아니라 Session, Room, Matchmaker 같은 객체와 연결될 수 있다.

예를 들어 Player가 현재 어떤 Room에 있는지, 어떤 Session과 연결되어 있는지, 매칭 중인지 플레이 중인지 같은 상태가 생긴다.

이때 클래스의 책임을 미리 나눠두지 않으면 RoomManager, SessionManager, GameManager가 서로의 일을 침범하기 쉽다.

클래스 다이어그램은 이런 객체들을 코드로 만들기 전에 한 번 펼쳐보는 용도로 쓸 수 있다. 어디까지가 Player의 책임인지, 어디부터는 Room이나 Session의 책임인지 생각해보게 된다.

✅ 마지막 요약

  • 클래스 다이어그램은 클래스의 구조와 관계를 보여주는 설계도다.
  • 코드 작성 전에 책임을 나누면 기능이 커져도 구조를 관리하기 쉽다.
  • GameManager 하나에 모든 기능을 넣으면 유지보수가 어려워진다.
  • Player와 Weapon처럼 역할을 나누면 객체지향 설계가 더 명확해진다.
  • UML은 예쁜 그림이 아니라 코드 구조를 생각하기 위한 도구다.

🧠 공부하면서 남긴 메모

클래스 다이어그램을 그린다는 건 코드를 그럴듯하게 포장하는 작업이 아니었다.

오히려 코드를 작성하기 전에 “이 데이터는 누구 것이지?”, “이 함수는 어느 클래스에 있어야 자연스럽지?”를 계속 물어보는 과정에 가까웠다.

Player가 공격한다는 말만 보면 Attack()을 Player에 두면 끝나는 것처럼 보인다. 하지만 실제로는 Weapon의 탄약을 줄여야 하고, 데미지를 계산해야 하고, 서버에서는 그 결과를 다른 클라이언트에게도 알려야 한다.

그 순간부터 설계가 필요해진다.

이번 예제에서는 Player와 Weapon만 나눴지만, 다음 단계에서는 Room, Session, Matchmaker까지 확장해서 클래스 후보를 뽑아볼 수 있을 것 같다.
